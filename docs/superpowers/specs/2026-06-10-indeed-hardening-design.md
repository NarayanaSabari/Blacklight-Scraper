# Indeed.com scraper — hardening + Monster-pattern lift

**Date:** 2026-06-10
**Scope:** Refactor `scrapers/indeed.js` to apply the same robustness pattern shipped for Monster + Dice: page-state classifier, typed errors, partial-result policy, file-backed cooldown for Cloudflare blocks, fixture-driven tests, debug harness. Existing partial hardening (`SCRAPER_STRICT_EMPTY` flag, `indeedNoResults` helper, `assertNotBlocked` block detector) is preserved; the strict-empty default flips from off to on.

## Goal

Stop the silent-failure modes still possible in the current Indeed scraper (`SCRAPER_STRICT_EMPTY=false` is the shipped default → 0-cards-on-200 looks identical to success), distinguish Cloudflare blocks from auth bounces from DOM rename, and prevent the cascade pattern Monster suffered when DataDome flagged the IP.

## Non-goals

- **No residential proxies.** Probe shows Cloudflare lets us through cleanly (5/5 at 2s spacing with `humanize:true`); not needed.
- **No persistent profile, no manual login.** Cookie-injection from the credentials API is the established pattern and works.
- **No changes to LinkedIn / Monster / Dice scrapers.** This slice ring-fences Indeed.
- **No changes to `BaseScraper`, `src/api/credentials.js`, or the registry beyond a one-line `strictEmpty:true` flip.**
- **No backend / orchestrator changes.** Cooldown is local to the host's filesystem.
- **No rewrite of the credential lease pattern.** Anonymous-fallback existed; this slice replaces it with `AuthError` to surface missing credentials loudly, but doesn't change how leases are acquired.

## Current scraper's documented gaps

The existing 783-line `scrapers/indeed.js` ships with `STRICT = process.env.SCRAPER_STRICT_EMPTY === 'true'` defaulting to `false`. The comment block explicitly says: *"When this is NOT 'true' (the default/shipped state) Indeed behaves byte-identically to the pre-1C scraper: loginSuccess set early, any 0-card page ends pagination, no block detection."*

Specifically, with strict off (today's default):
1. **No typed errors.** Failures log + return early. Untyped throws if anything bubbles.
2. **Silent zero-job paths.** A 0-card page ends pagination without distinguishing "no results for query" from "DOM changed" from "Cloudflare block" from "auth bounce."
3. **`assertNotBlocked` exists but is gated behind STRICT.** Block detection is opt-in.
4. **No cooldown.** Repeated Cloudflare blocks cascade — same pattern Monster suffered.
5. **No fixture-driven tests** beyond `indeed-block.test.js` (one helper).
6. **No debug harness** like `scripts/test-monster-scrape.js`.
7. **Auth-bounce is not distinguished from other failures.** `?start=10` → `secure.indeed.com/auth?` redirect produces an empty card count but no explicit `AuthError`.
8. **Pagination "loginSuccess set early"** — the code prematurely marks the session as authenticated, then can't recover gracefully if cookies are stale.

## Section A — Search-page extraction

**Per-page flow** (mirrors the Monster orchestrator's structure):

1. Build URL via existing `buildSearchUrl(domain, jobTitle, location, start)` (already handles parenthesis stripping, fromage=7, sort=date).
2. `page.goto(url, { waitUntil: 'load', timeout: 45000 })` (existing Cloudflare-passing tuning — keep).
3. `await sleep(CLOUDFLARE_GRACE_MS)` — 10 seconds (existing).
4. Capture `{url: page.url(), bodyText, anchorCount, sawAuthBounce, bytes}`:
   - `bodyText`: first 4KB of `innerText`.
   - `anchorCount`: `document.querySelectorAll('.job_seen_beacon').length || document.querySelectorAll('[data-jk]').length`.
   - `sawAuthBounce`: `/secure\.indeed\.com\/auth\?/.test(page.url())` — Indeed's auth-bounce signature.
   - `bytes`: `document.documentElement.outerHTML.length`.
5. Run `classifyIndeedSearchPage(...)` (see Section D).
6. Branch on the verdict.
7. On `results`: extract via Cheerio + existing `extractJobsFromSearchPage` logic.

**Selector waterfall** (preserve current behavior, document the data-jk migration):

```js
const cardSelectors = [
    '.job_seen_beacon',
    '[data-testid="job-card"]',
    '.resultContent',
    'a[data-jk]',  // (NEW — probe found data-jk migrated from li/div onto a)
    'li[data-jk]',
    'div[data-jk]',
];
```

The current waterfall has `'.jobsearch-ResultsList > li'` and `'li[data-jk]'`/`'div[data-jk]'` near the bottom — keep them as legacy fallbacks (probe shows they return 0 today, but they cost nothing to keep).

## Section B — Per-card extraction

DOM-only via Cheerio (Indeed has no structured data). Three pure helpers extracted from the current 783-line orchestrator so they can be fixture-tested:

- **`extractJobKey($card) → string | null`** — preserves the existing three-step fallback (card's own `data-jk`, closest ancestor `[data-jk]`, first child `[data-jk]`). Returns null if all three fail; caller skips the row.
- **`parseJobCard($card, domain) → {title, company, location, salary, postedDate, jobKey, url, ...} | {__domChanged: true, reason}`** — the row mapper. Returns the sentinel pattern Monster + Dice use when load-bearing fields are missing.
- **`indeedJobUrl(domain, jobKey) → string`** — constructs `https://${domain}/viewjob?jk=${jobKey}`. Used by the URL field and downstream apply-link.

Job-detail pages still go through the existing `extractJobDetails(page, job)` per-job navigation. That code stays as-is for this slice; it's not silently failing in practice and the rewrite scope already covers a lot.

## Section C — Drop the anonymous-fallback path

Today the orchestrator silently degrades to anonymous if the credential lease can't be acquired or cookies don't pass Cloudflare:

```js
// today (paraphrased)
if (!credential) { logProgress('Indeed', 'No credential — running anonymous'); }
```

That's a silent-degradation path: 16 cards/role instead of the expected 50-200, and operators don't notice.

Replace with:

```js
const credential = await getCredentialsAPIClient().acquire('indeed', sessionId);
if (!credential) {
    throw new AuthError('No Indeed credential available from API', { platform: 'indeed' });
}
```

The orchestrator handles `AuthError` via the existing typed-error path (BaseScraper records the failure, lease handles cooldown, operator gets a clear signal). If the operator genuinely WANTS anonymous mode (e.g. for testing), they can set `INDEED_ALLOW_ANONYMOUS=1` env which downgrades the throw to a warning. Default: throw.

## Section D — Page-state classifier `classifyIndeedSearchPage`

Pure function mirroring Monster's:

```js
classifyIndeedSearchPage({url, bodyText, anchorCount, sawAuthBounce, bytes}) → {state, signal}
```

States (resolution order):

1. **`soft_blocked`** — `/cloudflare|verify you are human|just a moment|ray id|additional verification|access denied/i.test(bodyText)` OR `url` matches `/captcha|challenge/`. Block-text wins over everything else (defensive — could mask cards in a stale prerender).
2. **`auth_required`** — `sawAuthBounce === true` (URL is `secure.indeed.com/auth?...page-two-signin`). NEW state, specific to Indeed.
3. **`results`** — `anchorCount > 0`.
4. **`empty_confirmed`** — `indeedNoResults(html)` returns true (existing helper, leverage as-is).
5. **`dom_changed`** — `anchorCount === 0`, no empty-text, no auth bounce, `bytes >= 50000` (page rendered fully but selectors stale).
6. **`network_error`** — fall-through (`anchorCount === 0`, body is small, no signals).

The `auth_required` state is the Indeed-specific addition. It surfaces as `AuthError` at the orchestrator level — distinct from Monster's `BlockedError` (which is the DataDome equivalent) because the remediation is different: rotate the credential, not wait for a cooldown.

## Section E — Typed errors + partial-result policy

Mirror Monster's pattern:

| Throw | When | Lease/cooldown |
|---|---|---|
| `AuthError({platform:'indeed'})` | classifier `auth_required` OR credential acquisition failed | Lease's `reportFailure(_, COOKIES_EXPIRED_COOLDOWN_MIN)` triggered (existing flow for LinkedIn). |
| `BlockedError({platform:'indeed', kind:'cloudflare'})` | classifier `soft_blocked` | Write `~/.blacklight-indeed-cooldown` marker; next call short-circuits at entry. |
| `DomChangedError({platform:'indeed'})` | classifier `dom_changed` (search-page); OR per-card extractor returns `__domChanged` on > 30% of cards in a batch | None — rethrow loud. |
| `NetworkError({platform:'indeed'})` | classifier `network_error`; OR `page.goto` throws | None. |

**Partial-result policy:** if a typed throw fires AFTER ≥ 1 job has been collected, return `{ jobs: collected, emptyConfirmed: false, partial: true }` instead of rethrowing. Same logic as Monster + Dice.

## Section F — Cooldown module `src/core/indeed-cooldown.js`

Mirror of `src/core/monster-cooldown.js`. Exports:

- `cooldownPath()` → `~/.blacklight-indeed-cooldown`
- `cooldownMs(env)` → 60-min default (matches Monster). Env override: `INDEED_BLOCK_COOLDOWN_MIN` (positive integer minutes).
- `readCooldownMarker({readFile, now, path})`, `writeCooldownMarker(...)`, `isOnCooldown(...)`
- `defaultReadFile() / defaultWriteFile() / defaultRename()`

**Why duplicate `monster-cooldown.js` instead of refactoring into a shared `scraper-cooldown.js`?** YAGNI for this slice — the duplication is ~45 LOC, the modules are isomorphic, and refactoring both would expand the scope and risk touching Monster code we just shipped. If we end up adding cooldown to a third platform (e.g. Glassdoor), parameterize it then.

### Orchestrator wiring (identical to Monster's pattern)

1. **Entry gate** at the top of `scrapeIndeed`: read marker, throw `BlockedError({kind: 'cloudflare-cooldown'})` if active. No browser launch.
2. **Write marker** before every `BlockedError` throw (the `soft_blocked` classifier branch) AND before every `network_error` throw (the cascade-prevention fix Monster needed too). The `auth_required` branch does NOT write the marker — that's a credential issue, not an IP issue.

## Section G — `strictEmpty: true` flip in the registry

Today:
```js
indeed: new BaseScraper('indeed', scrapeIndeed),
```

After:
```js
indeed: new BaseScraper('indeed', scrapeIndeed, { strictEmpty: true }),
```

`BaseScraper.execute` already handles `strictEmpty` (throws `BlockedError` on 0-jobs without `emptyConfirmed: true`). The scraper now returns `{jobs: [], emptyConfirmed: true}` for `empty_confirmed` verdicts so the strict gate doesn't false-positive.

## Open questions (deferred to implementation discovery)

1. **Job-detail page extraction** — the current 783-line scraper has `extractJobDetails(page, job)` for per-job navigation. The probe didn't deep-dive on detail pages. If the per-job extraction starts silently failing, that's a follow-up slice. For now, preserve current behavior.
2. **Sponsored vs organic cards** — `[data-empn]` selector found 9 sponsored cards in the probe; the current code doesn't distinguish them. Should we flag `isPromoted: true` on sponsored rows (like Monster does)? Cheap; do it.
3. **Regional domain handling** — `getIndeedDomain(location)` (existing) returns `in.indeed.com` for Indian locations etc. Probe used `www.indeed.com`. Verify regional domains pass Cloudflare during implementation; same humanize tuning should work.

## Success criteria

- `scripts/test-indeed-scrape.js -- "software engineer"` returns ≥ 30 jobs (1-page minimum if credential acquires; up to 200 across 5 pages if pagination works), every row has non-empty title + company + URL starting with `https://www.indeed.com/viewjob?jk=<key>`. **No `title === 'N/A'` rows**.
- A captured Cloudflare interstitial fed through `classifyIndeedSearchPage` returns `soft_blocked` → `BlockedError` → cooldown marker written → next call short-circuits in < 10ms.
- A captured `secure.indeed.com/auth?` redirect URL returns `auth_required` → `AuthError` thrown. Marker NOT written.
- The no-results fixture returns `empty_confirmed` and emits `{jobs: [], emptyConfirmed: true}` (no false strict-empty alarm).
- `scraper_url_quality_total{platform="indeed",quality="permalink"}` ticks for every job.
- `scrapers/linkedin.js`, `scrapers/monster.js`, `scrapers/dice.js` not modified (sanity guard).
