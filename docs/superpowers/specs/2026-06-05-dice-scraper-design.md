# Dice.com scraper — rewrite design

**Date:** 2026-06-05
**Scope:** Rewrite `scrapers/dice.js` from a working-but-silent scraper into a hardened one (typed errors, page-state classifier, fixture tests, debug harness) that keeps the Schema.org JSON-LD path as the canonical extraction surface and drops two dead features (recruiter, easyApply) that 10/10 probe samples showed are no longer populated.

## Goal

Same robustness bar as the Monster slice: when Dice works, return clean job rows; when Dice breaks (DOM rename, JSON-LD removal, parse error, anti-bot), throw a typed error so the orchestrator can route it. Stop silently emitting `'N/A'` for fields Dice no longer ships.

## Non-goals

- **No anti-bot infrastructure.** Probe shows 10/10 success at zero spacing; Dice has no detection layer worth defeating. `humanize:false` stays.
- **No persistent profile, no login.** Dice exposes job content to anonymous traffic.
- **No proxy rotation.** Not needed.
- **No backend changes.** Same job-row envelope as every other scraper.
- **No rebuild of the 2-stage architecture** (search → 100 detail pages in parallel). The structure is sound; we only harden the failure modes around it.

## Current scraper's documented gaps (rewrite addresses each)

The existing 361-line `scrapers/dice.js` works on the happy path but every failure path silently degrades to "0 jobs" or "field N/A":

1. **No typed errors** — every catch logs and `return`s or `break`s.
2. **No `strictEmpty:true`** — 0-jobs-on-200 looks identical to "no results."
3. **Search loop break-on-error** — page 2 timeout silently truncates to page 1.
4. **Detail parser silent drop** — `script[id="jobDetailStructuredData"]` missing → log + `return`; per-job loss invisible.
5. **Recruiter fetch dead feature** — `recruiterId` regex returns 0/10 in current production; every Dice job in the DB has `recruiter.name = 'N/A'`.
6. **`easyApply` regex dead** — same 0/10 in current production.
7. **No tests** — `test/scrapers/` has zero Dice files; no fixture, no harness, no regression guard for the structured-data shape.
8. **No page-state classifier** — can't disambiguate "no results for query" / "DOM changed" / "anti-bot."
9. **Per-job recruiter navigation** — even when recruiterId IS found, the second goto adds ~2-3s per job for an output field that downstream consumers can't rely on anyway.

## Section A — Search-page extraction

**Per-page flow:**

1. `page.goto(searchUrl, {waitUntil:'domcontentloaded', timeout:30_000})` where `searchUrl = https://www.dice.com/jobs?q=<role>&location=<loc>&filters.postedDate=SEVEN&page=<N>`.
2. `await page.waitForSelector('a[href*="/job-detail/"]', {timeout:5000})` — soft signal; classifier owns the verdict.
3. Read `{bodyText, anchorCount, sawSearchAnchors}` via `page.evaluate`.
4. Run `classifyDiceSearchPage(...)`.
5. Branch on the verdict (see Section D).
6. On `results`: collect anchors via primary selector `a[href*="/job-detail/"]`; if 0, fall back to `[data-testid*="job-card"] a[href*="/job-detail/"]`; dedupe.
7. No artificial pacing between search-page navigations — probe shows zero benefit.

**Stop conditions** (same priority order as Monster):
- `pageNum > MAX_PAGES` (5)
- collected URLs ≥ `MAX_JOBS` (100)
- 2 consecutive `empty_confirmed` pages
- ANY typed throw if no URLs collected yet; partial return if some URLs collected

## Section B — Detail-page extraction (Schema.org JSON-LD)

Each `/job-detail/<id>` URL → render in CloakBrowser (10 concurrency, 5 contexts as today) → parse:

```js
const scriptEl = document.querySelector('script[id="jobDetailStructuredData"]');
if (!scriptEl) → DomChangedError('jobDetailStructuredData script missing', {platform:'dice', url})
const raw = scriptEl.textContent;
let jsonLd;
try { jsonLd = JSON.parse(raw); }
catch (e) { → ParseError('jobDetailStructuredData JSON parse failed', {platform:'dice', cause:e, url}); }
if (jsonLd?.['@type'] !== 'JobPosting') → DomChangedError(`expected JobPosting, got ${jsonLd?.['@type']}`, {platform:'dice', url});
```

But: a single bad detail page must NOT take down a 100-job batch. The orchestrator's per-job task catches `DomChangedError`/`ParseError`/`NetworkError`, records the failure into a per-batch counter, AND skips the row. Threshold: if > 30% of detail pages throw `DomChangedError`/`ParseError`, the BATCH throws `DomChangedError` once at the end (after the crawler finishes) so an extractor-level break is loud.

**Pure helpers exported alongside `scrapeDice`:**

- `parseStructuredData(scriptText) → {data, error}` — JSON.parse wrapper, returns `{data: parsed, error: null}` on success or `{data: null, error: msg}`.
- `extractJobFromStructuredData(jsonLd, requestUrl) → {jobId, title, company, locationFormatted, city, state, country, isRemote, salaryFormatted, salaryMin, salaryMax, salaryCurrency, salaryPeriod, employmentType, postedDate, validThrough, description, url, companyProfileUrl, companyLogoUrl}` — pure mapping function. Throws (via return-of-`__domChanged` sentinel) when `title` or `hiringOrganization.name` is missing.
- `parseSalary(baseSalary) → {min, max, currency, period, formatted}` — handles `MonetaryAmount` shape today, `value.minValue` legacy fallback, `formatted` string with `/yr` or `/hr` suffix when period is known.
- `parseEmploymentType(rawType) → 'full_time'|'part_time'|'contract'|'temporary'|'internship'|<raw lower>` — handles single string or array.
- `extractSkills(cheerioOrDom) → string[]` — finds the `Skills` heading then walks its sibling `<ul>` for `<li>` skills.
- `extractWorkplaceType(cheerioOrDom) → string|null` — reads `[data-testid="locationTypeBadge"]` text.

Skills + workplace type continue to use Cheerio in the existing CheerioCrawler context (no change to that data-flow).

## Section C — Drop recruiter + easyApply (dead features)

The 10-sample recruiter probe found 0/10 hits for both the `recruiterId` regex and the `easyApply` regex. Their data is no longer in the page payloads we can read. Behavior change:

- **Remove `fetchRecruiterProfile`** function entirely. Saves one cross-page navigation per job (~half the per-job cost).
- **Remove the `recruiterId` regex sniff** in the detail handler.
- **Remove the `easyApply` regex sniff** in the detail handler.
- Drop `recruiter` and `easyApply` from the `normalizeJobData(...)` input — the downstream normalizer will simply not have those fields. (Confirm no schema regression: a quick check on the normalizer shows `recruiter` is already conditionally pruned when missing; same for `easyApply` if it's never set.)

This is the biggest LOC reduction in the slice (~70 LOC out, faster runtime, removes a code path that's lying to downstream).

## Section D — Page-state classifier `classifyDiceSearchPage`

Pure function mirroring Monster's:

```js
classifyDiceSearchPage({url, bodyText, anchorCount}) → {state, signal}
```

- `results` — `anchorCount > 0`
- `empty_confirmed` — `/no jobs found|no results|0 jobs match/i.test(bodyText)` AND `anchorCount === 0`
- `soft_blocked` — `/cloudflare|access denied|please verify|ray id/i.test(bodyText)` OR `url` matches `/captcha|challenge/` (defensive — probe showed 0/10 blocks; keep cheap)
- `dom_changed` — `anchorCount === 0`, no empty/block text, page rendered ≥ 50KB (real-looking page just missing what we expect)
- `network_error` — fall-through (rendered page < 50KB and no signal — treat as transport-level failure since something clearly went wrong)

Body-text length threshold (`50_000`) is the tie-breaker between "DOM rendered but cards gone" (dom_changed) and "page never loaded properly" (network_error). 50 KB is conservative — the current search page is ~360 KB.

## Section E — Error model + partial-result policy

Reuse `src/core/errors.js` types — same as Monster:

| Throw | When | Cooldown |
|---|---|---|
| `BlockedError({platform:'dice',kind:'cloudflare'})` | classifier returns `soft_blocked` | 30 min |
| `DomChangedError({platform:'dice'})` | search-page selectors return 0 AND not empty_confirmed/soft_blocked; OR > 30% of detail pages throw structured-data errors in a batch; OR `parseStructuredData` returns `error` for the structured-data script | none — rethrow loud |
| `ParseError({platform:'dice'})` | structured-data `JSON.parse` fails | none |
| `NetworkError({platform:'dice'})` | `page.goto` timeout / classifier `network_error` | none |
| Per-detail-row catch | bad row skipped, counter incremented; threshold check at batch end | n/a |

**Partial-result policy** mirrors Monster: if a typed throw fires AFTER ≥ 1 job has been collected, return `{jobs: collected, emptyConfirmed: false, partial: true}` instead of rethrowing. If 0 jobs at throw time, propagate.

**`strictEmpty: true`** wired in the registry so `BaseScraper.execute` throws `BlockedError` when `scrapeDice` returns `[]` without `emptyConfirmed: true`.

## Open questions (deferable to implementation discovery)

1. **`extractSkills` location post-rewrite** — does it stay inside the page-bound Cheerio handler or get extracted as a pure helper? Lean toward pure with the Cheerio object passed in, so the test can drive it from a static fixture. Confirm during implementation.
2. **Concurrency cap** — current scraper uses `maxConcurrency: 10` on the detail crawler. The probe shows zero anti-bot, so 10 is fine. If the orchestrator's overall fleet hits ~50 Dice scrapes/hr that's still trivial for Dice. Leave at 10.
3. **`identifier.value` vs URL-trail** — current scraper falls back to `request.url.split('/').pop()` when `identifier.value` is missing. The probe confirmed `identifier` is present in JSON-LD (`{@type:'PropertyValue', name:'<company>', value:'<uuid>'}`); keep the fallback.

## Success criteria

- `scripts/test-dice-scrape.js -- "software engineer"` returns ≥ 30 jobs with **every** row having: non-empty title (length > 1), non-empty company, URL starting with `https://www.dice.com/job-detail/`, no `'N/A'` placeholders. **PERMALINK rate = 100%.** No `recruiter.name === 'N/A'` rows (because the field is dropped entirely).
- A captured detail page with the structured-data script DELETED, fed through `parseStructuredData`, returns the error path → `DomChangedError`.
- A captured detail page with `@type:"Article"` (forged), fed through `extractJobFromStructuredData`, throws `DomChangedError`.
- A captured search page with 0 anchors but `/no jobs found/` body text returns `empty_confirmed`.
- A captured Cloudflare interstitial page returns `soft_blocked` → `BlockedError`.
- `scraper_url_quality_total{platform="dice",quality="permalink"}` ticks for every job; `quality="empty"` is 0.
- `scrapers/linkedin.js` and `scrapers/monster.js` not modified (different platforms — sanity guard).
