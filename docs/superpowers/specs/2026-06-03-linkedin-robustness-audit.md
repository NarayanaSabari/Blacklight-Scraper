# LinkedIn scrape workflow — robustness audit (2026-06-03)

Multi-agent audit. 39 subagents, 6 dimensions, P0/P1 findings adversarially re-verified. 62 confirmed findings (1 P0, 17 P1, 44 P2+).

## Executive summary

The LinkedIn scraper is functionally correct on the happy path (commit 5061ace verified 8/8 live permalinks) but is fragile and largely unobservable — a sustainable production system needs work along three axes before the next LinkedIn-side change. The single biggest weakness is silent failure: the permalink resolver returns empty strings on every error branch, no metric records the result, BaseScraper records by jobs.length without inspecting url quality, and the boot log carries no SHA/profile/headless signal — so a 100%-empty-job_url regression (whether caused by a stale `node server.js` post-`git pull`, a LinkedIn aria-label A/B flip, an authwall, or a logged-out profile) looks identical to a healthy scrape in logs, metrics, and the / health endpoint. Four cross-cutting themes account for ~60% of the findings: (1) no observability of the load-bearing permalink resolver (no rate metric, no per-branch logging, no contract test, no boot SHA, no /healthz with auth state); (2) lease/session lifecycle races (lease leaks on launch failure, reportSuccess forgets the lease before reportFailure can cool it, reestablish isn't single-flighted and closes peer pages mid-scrape, no zombie-context recovery, no PID-tracking SIGKILL on shutdown); (3) single-source brittleness in the scrape DOM (hard-coded English aria-label, no permalink fallback waterfall, no componentkey DOM canary, STRICT-gated 0-post throw defaults off); (4) setup/deployment lies (wizard claims success without verifying the persistent profile is logged in, docs warn against the restart that is actually mandatory after git pull). All four themes feed back into the same operational reality: when something breaks, operators learn from downstream data quality, not from the running scraper.

## P0 — must fix

### Hard-coded English aria-label is single point of failure for ALL permalinks

- **Category / effort:** scrape / S
- **What breaks:** A single LinkedIn aria-label flip (e.g. 'Open control menu' → 'Post control menu' → 'More options for post by') causes 100% empty job_url on the NEW DOM cohort with zero operator signal. The legacy-DOM timestamp-anchor fallback exists at :1003-1016 but was never carried into the NEW DOM extractor. Combined with the silent-success path (resolver returns '' → postUrl='' → BaseScraper still records the row as success), this is data-quality collapse masquerading as healthy throughput. This is also the proximate failure mode the audit was commissioned for.
- **Evidence:** scrapers/linkedin.js:251 uses sole selector `button[aria-label^="Open control menu for post by"]`; same string is the author-name fallback at :1084-1088; CloakBrowser forces locale en-US (:192, :234) but LinkedIn A/B-tests aria-label wording independent of locale; the NEW DOM extractor at :1128-1150 has no other source for the activity URN, so the menu click is the ONLY way to recover it for the dominant DOM path; resolvePostUrlViaMenu (:246-264) silently returns '' on any failure; no metric, no canary, no fallback selector.
- **Fix:** Two-layer fix in scrapers/linkedin.js: (1) Compound selector tolerance at :251 — try `button[aria-label^="Open control menu"], button[aria-label*="control menu for post"], button[aria-label*="More options"], button.feed-shared-control-menu__trigger, button[data-test-control-menu-trigger]`. Same for the author-name regex at :1084-1088. (2) Permalink waterfall in resolvePostUrlViaMenu (:246-264) — keep the report-in-modal poll but widen to `a[href*="updateUrn=urn:li:activity"]` (route-name independent), then fall back to the post container's timestamp anchor (the pattern the legacy extractor already uses at :1003-1016), then scan inline `<script>` text in the post container for `urn:li:activity:\d+`. Scope all queries to the post container handle `el` (already captured at :249) — never `document.querySelector`. Add `linkedin_menu_selector_zero` and per-step `permalink_resolved_via=<step>` metrics so the next rotation is visible within one run.

### #establish leaks lease if launchPersistentProfile throws

- **Category / effort:** session / S
- **What breaks:** On a host where CloakBrowser is broken, each failed boot leaks 1 credential. With 10 retries the orchestrator can leak 10 credentials per startup attempt. Over crashes this starves the credential pool.
- **Evidence:** src/scrapers/linkedin-session.js:37-46 — `this._lease = lease` (line 40) is assigned BEFORE awaiting this._launch() (line 44).
